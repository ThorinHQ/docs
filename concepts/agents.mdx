Agents are the actors in Thorin. Each agent watches one or more mailboxes, follows rules you define, and uses its tools to propose or take action on email. This page covers the two kinds of agents and the pieces you configure on each one.

## Personal vs. shared agents

Thorin has two kinds of agents:

- **Personal agent** — every user gets one. It reads that user's mailbox and acts on their behalf. You can't have more than one personal agent per user.
- **Shared agents** — act on behalf of a process, a team, or a shared mailbox like `support@` or `ops@`. A shared agent can have multiple human owners and is usually scoped to a specific responsibility.

Most users start with their personal agent and add shared agents as the team formalizes a process worth automating.

<Tip>
  If your admin has designated an [Email Agent](/integrations/connectors#email-agent) mailbox, you can reach your personal agent by emailing — or CC'ing — that address from any thread. Thorin replies from the same address and follow-ups on the thread continue the same session.
</Tip>

## What's on an agent

Every agent — personal or shared — is configured with the same set of pieces:

- **Name** — how the agent is identified across Thorin.
- **Owners** — the humans who can edit the agent's configuration. Personal agents are owned by the user they belong to; shared agents can have several owners.
- **Delegate** — an optional user the agent acts on behalf of. The delegate's identity is used when the agent needs to send mail or take action that requires a human in the loop.
- **Contacts** — the external recipients the agent is allowed to communicate with, plus the permission grants that say what kinds of messages it can send each one.
- **Resources** — documents, entities, and integrations the agent is allowed to read or call.
- **Skills** — reusable capabilities installed on the agent (see [Skills](/features/skills)).
- **Responsibilities** — the standing jobs the agent is expected to handle.
- **MCP servers** — additional tool servers the agent can call, including [personal MCPs](/integrations/personal-mcps) on your personal agent.
- **Report subscription** — the recurring summary of what the agent did, delivered to subscribers.

## Where to find agents

Open **Agents** in the sidebar to see every agent you can access. You can also navigate to an agent from **Library** > Projects when the project lists the agent as a participant.

## Contact permissions

By default an agent's outbound communication is restricted to a list of allowed contacts. Each contact carries a permission type that controls what kinds of messages the agent can send them — for example, replying to a thread the contact started versus initiating a new outreach.

<Note>
  If an agent needs to message someone who isn't on its contact list, an owner has to add the contact and pick a permission grant first.
</Note>

## Sessions

Every time an agent runs in response to a trigger — an incoming email, a scheduled job, a chat message — it produces a **session**. Sessions are visible in the agent's history panel and capture the input that triggered the run, the agent's reasoning, and the actions it took or proposed.

Use sessions to audit past behavior, debug a surprising decision, or copy a successful run into a new automation.

## Example: edit owners and delegate

<Steps>
  <Step title="Open the agent">
    Go to **Agents** in the sidebar and pick the agent you want to edit. You need to be an owner to change configuration.
  </Step>
  <Step title="Update owners">
    In the agent's settings, add or remove users from the **Owners** list. Owners can edit any part of the agent.
  </Step>
  <Step title="Set the delegate">
    Pick a user as the **Delegate**. The agent will act on behalf of that user when it sends mail or takes other identity-bound actions.
  </Step>
  <Step title="Save">
    Save the changes. The next session the agent runs will use the new owners and delegate.
  </Step>
</Steps>

<Tip>
  Keep the delegate aligned with whoever is accountable for the work the agent does. For shared agents that handle inbound support, the delegate is usually the team lead.
</Tip>
