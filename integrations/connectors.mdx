First-party connectors are integrations Thorin builds and maintains directly, separate from the [MCP](/integrations/mcps) catalog. An admin configures each one once from **Admin → Integrations**, and it becomes available across the org.

This page covers what each connector adds and the headline auth requirement. Configuration UI evolves over time — see the [Integrations overview](/integrations/overview) for the current list of surfaces.

## Affinity

Pulls CRM records from your Affinity workspace so agents can read accounts, organizations, and people without leaving an email thread.

- **Where to configure:** Admin → Integrations → Affinity.
- **Auth:** Affinity API key, generated from your Affinity admin settings and pasted into the connector dialog.

## Gong

Surfaces call transcripts and metadata so agents can reference what was actually said on a call when answering follow-up email.

- **Where to configure:** Admin → Integrations → Gong.
- **Auth:** Gong API keys (access key and secret), issued from your Gong admin panel.

## Google Drive

Lets agents read documents from a shared Drive — playbooks, briefs, contracts, anything you want them to cite.

- **Where to configure:** Admin → Integrations → Google Drive.
- **Setup:**
  1. Create a shared Drive in Google Workspace.
  2. Grant Thorin's system user access to that Drive.
  3. Paste the Drive link into the connector dialog.
- **Auth:** Workspace-level access via the system user — no per-user OAuth.

## Salesforce

Read-only access to Salesforce records — accounts, opportunities, contacts — scoped to each user's own Salesforce permissions.

- **Where to configure:** Admin → Integrations → Salesforce.
- **Setup:** Provide a Salesforce **Client ID** from your Salesforce connected app.
- **Auth:** Per-user OAuth. Each teammate authorizes their own Salesforce account; Thorin reads only what that user can read.

<Tip>
  If you also want Salesforce write access through MCP, install the **Salesforce** entry from the [MCP catalog](/integrations/mcps#add-from-the-catalog) alongside the first-party connector.
</Tip>

## Email Agent

Designates a single tenant mailbox that Thorin is allowed to send email **as** on behalf of users' personal agents. Once configured, anyone in your tenant can email — or CC — that address from any thread to reach their own personal agent, and Thorin replies from the same address, threaded under the original message.

Use a dedicated mailbox for this (for example, `thorin@your-domain.com`) — not an individual teammate's personal inbox.

- **Where to configure:** Admin → Integrations → Email Agent.
- **Prerequisites:**
  - The mailbox address must be on your organization's domain (the same domain as the admin designating it). Thorin rejects addresses on any other domain.
  - Thorin must already have **domain-wide delegation** access to send as that mailbox. Designation runs a read-only Gmail probe against the address before saving; if delegation isn't granted, the designation is refused and nothing is persisted.
- **One per tenant:** only one active email agent service account exists at a time. Designating a new address replaces the previous one.
- **Auth:** Google Workspace domain-wide delegation, set up alongside the other [mail-access options](/admin/onboarding) during onboarding.

### Designate the mailbox

1. Open **Admin → Integrations** and find the **Email Agent** card.
2. Click **Designate** and enter the mailbox address (for example, `thorin@your-domain.com`).
3. Confirm the authorization prompt — Thorin will be able to send email as this address on behalf of your users.

If the access probe fails (Thorin doesn't have permission to act as the address), the dialog surfaces an error and no service account is saved.

### How users reach their agent

Once a mailbox is designated, any user in the tenant can reach their [personal agent](/concepts/agents) by:

- Sending a new email to the service account address with their question in the subject or body.
- Adding the service account address to the `To` or `Cc` line of an existing thread when they want their agent involved.

Thorin replies from the service account address, threaded under the user's message. Each reply runs the sender's own personal agent with that user's allowed context, integrations, and skills — a teammate CC'ing the service account on your thread does not give them access to your agent.

To continue a conversation, the user keeps replying on the thread. Thorin resumes the same agent session, reads only the messages added since its last reply, and answers in line. The session is reused across turns so the agent doesn't lose context between replies.

The channel is **tenant scoped**:

- Only senders who are members of your Thorin tenant trigger a response. Messages from external senders, and messages sent from the service account mailbox itself, are ignored.
- Every inbound message — subject, body, and quoted thread — is scanned for prompt-injection content before being passed to the agent.

### Remove the mailbox

From the **Email Agent** card, click **Remove** and confirm. Thorin will stop sending email as that address immediately. You can designate a new mailbox at any time.

<Tip>
  Designation, removal, and every reply Thorin sends through the channel are recorded in [audit logs](/admin/audit-logs) under the integration's resource type — including the actor, the affected address, and the thread the reply belongs to.
</Tip>

## Slack

Lets agents post to Slack channels, read recent context, and respond when mentioned.

- **Where to configure:** Admin → Integrations → Slack.
- **Setup:**
  1. An admin installs the Thorin Slack app into the workspace.
  2. Add the Thorin bot to each channel you want it active in.
- **Auth:** Slack workspace OAuth, granted by the admin during install.

Channel reachability follows Slack's normal rules:

| Channel type | Reachable? |
| --- | --- |
| Public channel | Yes — the bot can join any public channel. |
| Private channel | Only if explicitly invited. |
| DM | Only if a user opens a DM with the bot. |
