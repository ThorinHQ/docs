Thorin reaches outside email through two complementary surfaces: a small set of **first-party connectors** an admin configures once for the whole org, and **MCP servers** that each user adds for themselves using the open [Model Context Protocol](https://modelcontextprotocol.io) standard. Together they're how agents read CRM records, post to Slack, file Linear issues, search Notion, and everything in between.

## Where integrations live

- **Admin → Integrations** — admin-managed first-party connectors. Anything here is available to every user in the tenant once they authorize their own connection.
- **Chat composer (+ → Connectors)** — **personal MCPs** that each user adds to their own personal agent. See [Personal MCPs](/integrations/personal-mcps).

## First-party connectors

Purpose-built integrations Thorin maintains directly. An admin configures each one once, and it becomes available across the org.

| Connector | What it adds |
| --- | --- |
| **Affinity** | CRM records pulled from your Affinity workspace. |
| **Gong** | Call transcripts and metadata. |
| **Google Drive** | Document access through a shared Drive. |
| **Salesforce** | Read-only Salesforce records, scoped per user. |
| **Slack** | Bot access to channels you invite Thorin into. |

See [First-party connectors](/integrations/connectors) for setup details.

## Personal MCPs

For everything else, Thorin speaks Model Context Protocol. MCP is the standard most modern AI tools use to expose capabilities, and any MCP-compatible service can plug into Thorin without custom integration work.

Personal MCPs are added by individual users from the **+** menu in the chat composer. The connectors dialog ships with one-click presets — currently Notion, Linear, GitHub, Granola, Sentry, HubSpot, Figma, and Salesforce — and supports adding any other server by URL. See [Personal MCPs](/integrations/personal-mcps) for the full flow.

<Note>
  MCPs in Thorin are personal-only. Admins do not install MCP servers for the whole tenant — each user adds the servers they want themselves.
</Note>

## Authentication

Most MCP servers authenticate over OAuth. Thorin uses [Dynamic Client Registration (RFC 7591)](https://datatracker.ietf.org/doc/html/rfc7591) so users don't have to copy client IDs and secrets by hand — Thorin registers itself with the server on the fly and runs an authorization code + PKCE flow. After sign-in, you're returned to the chat you launched from with a success or error toast.

For servers that don't support OAuth dynamic client registration, paste a long-lived **static access token** as a fallback. Thorin uses it as a bearer credential when calling the server.
